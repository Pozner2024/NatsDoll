# Внешняя страница оплаты на WooCommerce (pay.natsdoll.com)

Дата: 2026-07-03
Статус: дизайн утверждён, ждёт implementation-плана

## Контекст и цель

Посредник «Склад ЮСА» (его PayPal-аккаунт принимает деньги покупателей) не может выдать
Client ID/Secret: их техотдел умеет подключать PayPal только через onboarding-флоу
«Connect to PayPal» (вход в аккаунт), доступный платформам-партнёрам PayPal —
WooCommerce, Shopify и т.п. Кастомному сайту этот флоу недоступен.

Решение: постоянная внешняя страница оплаты на WordPress + WooCommerce на поддомене
`pay.natsdoll.com` (тот же VPS, место докупается). PayPal посредника подключается к
WooCommerce его собственным способом («Connect to PayPal»), а наш сайт взаимодействует
с WooCommerce программно.

Отклонённые альтернативы:

- Уговаривать посредника выдать ключи — пользователь решил не давить.
- Извлечь ключи из одноразового WooCommerce после Connect — пользователь не хочет
  рисковать (серые условия использования).
- Shopify — платный, риск блокировки аккаунта из Беларуси, меньше контроля.
- PayPal.Me — нет надёжной гостевой оплаты картой, ручная сверка каждого платежа.
- Legacy-кнопки WPS/_xclick — PayPal отключает их к январю 2027.
- WooCommerce как полноценный чекаут — дублирование каталога и адресной формы.
- Поллинг статуса Woo-заказа вместо вебхука — задержки, лишние запросы, вебхук-паттерн
  в проекте уже есть.

## Утверждённые решения

1. Существующий PayPal-модуль (server-flow и client-flow, встроенные кнопки) не
   трогаем. Внешняя страница — третий режим, переключается в админке.
2. После оплаты покупатель автоматически возвращается на нашу страницу подтверждения
   заказа.
3. Заказ-двойник в Woo содержит полный состав: каждый товар строкой с названием и
   ценой, отдельно доставка.
4. Оформление страницы оплаты — лёгкий брендинг поверх минимальной темы: логотип
   NatsDoll, фирменные цвета/шрифт через CSS. Без разработки полноценной темы.

## Архитектура

### Инфраструктура

- `docker-compose.prod.yml`: два новых сервиса — `wordpress` (официальный образ) и
  `wp-db` (MariaDB; WordPress не поддерживает PostgreSQL — жёсткое требование
  платформы). Свои volumes, своя БД, Postgres основного сайта не затрагивается.
- `Caddyfile`: новый хост-блок `pay.natsdoll.com` по образцу `stats.natsdoll.com` —
  reverse proxy на `wordpress`, свои security-заголовки (CSP с доменами PayPal),
  `X-Robots-Tag: noindex, nofollow` на все ответы.
- DNS: A-запись `pay` → 89.127.205.44 добавляет пользователь на Namecheap.
  Сертификат Caddy выпускает автоматически.
- Локальная разработка: те же сервисы в dev-compose (детали — в implementation-плане).

### WordPress / WooCommerce

- Плагины: WooCommerce + WooCommerce PayPal Payments. Минимум других плагинов.
- Гостевой чекаут включён; регистрация покупателей выключена; все письма WooCommerce
  покупателям выключены (письма шлёт только основной сайт).
- «Discourage search engines» включён (в дополнение к X-Robots-Tag).
- Автообновления WP и плагинов включены.
- Мини-плагин (mu-plugin) для авто-редиректа со страницы «спасибо» обратно на
  страницу подтверждения заказа natsdoll.com (URL возврата приходит в meta заказа).
- PayPal посредника подключается кнопкой «Connect to PayPal» в настройках плагина —
  одноразовое действие в wp-admin, покупатели этого никогда не видят. До подключения
  посредника плагин работает с нашим собственным sandbox-аккаунтом (ручной ввод
  ключей в advanced-настройках) — весь флоу тестируется без участия посредника.

### Настройки и данные

- `.env` (новые переменные): `WOO_BASE_URL`, `WOO_CONSUMER_KEY`, `WOO_CONSUMER_SECRET`
  (REST-ключи из админки Woo, scope Read/Write), `WOO_WEBHOOK_SECRET`. Ключи наши,
  меняются практически никогда — храним в env, не в БД.
- Настройки платежей (админка сайта, существующая секция): новый переключатель
  «External payment page (WooCommerce)». Одна новая колонка в таблице настроек
  платежей + миграция. `GET /api/payments/config` отдаёт признак внешнего режима;
  при включённом внешнем режиме фронт показывает кнопку Pay вместо PayPal-кнопок.
- `Order` (Prisma): два новых поля — `wooOrderId Int?` и `wooOrderKey String?`
  (ссылка на оплату строится из них; раздельное хранение переживает смену домена).

## Поток данных

1. Корзина → `prepareOrder()` как сейчас → заказ `PENDING` (авторизованный или гость).
2. Фронт зовёт новый `POST /api/payments/woo/create-payment { orderId }`:
   - у заказа уже есть `wooOrderId` → вернуть существующую ссылку (идемпотентность,
     повторные нажатия не плодят дубликаты);
   - иначе создать заказ в Woo через REST API: состав построчно, строка доставки,
     billing-адрес из нашего заказа, метка `natsdoll-<orderNumber>`, URL возврата;
     сохранить `wooOrderId`/`wooOrderKey`; вернуть ссылку на страницу оплаты.
3. Фронт: `window.location.href` → страница оплаты Woo. Покупатель ничего не
   заполняет — видит состав, сумму и кнопки: жёлтая PayPal и тёмная
   «Debit or Credit Card» (гостевая оплата картой без PayPal-аккаунта).
4. После оплаты mu-plugin редиректит на нашу страницу подтверждения заказа.
5. Параллельно Woo шлёт вебхук `POST /api/payments/woo/webhook` (смена статуса
   заказа). API проверяет подпись и сумму, находит заказ по `wooOrderId`,
   вызывает существующий `markOrderPaid` → `PAID`, сток списывается тем же
   механизмом, что при PayPal-capture.
6. Страница подтверждения: заказ ещё `PENDING` → «Payment is being processed…» +
   короткий поллинг статуса до прихода вебхука (обычно секунды).
7. Неоплаченный заказ остаётся `PENDING`; из кабинета доступна кнопка Pay — тот же
   эндпоинт возвращает ту же ссылку.

## Ошибки и краевые случаи

- Woo недоступен при создании платежа → toast с понятной ошибкой, заказ остаётся
  `PENDING`, повторное нажатие Pay допустимо.
- Сумма в вебхуке ≠ `totalAmount` заказа → `PAID` не ставим, пишем в лог для
  ручного разбора.
- Дубликаты вебхуков → `markOrderPaid` идемпотентен (уже реализовано).
- Наш заказ отменён, Woo-двойник висит → отмена в админке Woo вручную (YAGNI).
- Refund → вручную, как сейчас.

## Безопасность

- Вебхук: проверка `X-WC-Webhook-Signature` (HMAC-SHA256 от сырого body,
  секрет `WOO_WEBHOOK_SECRET`), сравнение constant-time, rate limit — по образцу
  существующего PayPal-вебхука.
- REST-ключи Woo только в `.env`, не в git.
- WP hardening: xmlrpc выключен, сильные пароли, минимум плагинов, автообновления.
  Доступ посредника в wp-admin — временный, только на момент «Connect to PayPal»,
  после подключения учётка отключается.
- Поддомен не индексируется (X-Robots-Tag + настройка WP): SEO основного сайта не
  затрагивается, дублей контента нет.

## Тестирование

- Unit (api): use-case создания платежа (создание/переиспользование ссылки, Woo
  недоступен) и обработки вебхука (подпись, сумма, идемпотентность, неизвестный
  wooOrderId) — mock-клиент Woo, по образцу существующих PayPal-тестов.
- Unit (web): кнопка Pay в внешнем режиме (redirect), поллинг на странице
  подтверждения.
- Ручной e2e до посредника: наш sandbox PayPal в Woo-плагине, полный путь
  корзина → оплата (PayPal и гостевая карта) → `PAID`.
- После подключения посредника: live-платёж на пару долларов, проверка поступления
  в их кабинете и статуса `PAID` у нас.

## Порядок внедрения

1. WP + Woo на VPS (compose, Caddy, DNS, брендинг, sandbox-ключи) — флоу оплаты
   проверяется вручную внутри Woo.
2. API-интеграция: клиент Woo, create-payment, вебхук, миграции, тесты.
3. Фронт: режим external в корзине/кабинете, страница подтверждения с поллингом.
4. Прод-деплой, e2e на sandbox.
5. Письмо посреднику, «Connect to PayPal», переключение в live, контрольный платёж.

## Вне скоупа

- Автоматическая отмена/refund Woo-заказов.
- Синхронизация каталога с Woo (в Woo нет товаров — только заказы-двойники).
- Удаление существующих PayPal-режимов (остаются как есть).
