# Гостевая оплата (checkout без регистрации)

Дата: 2026-06-27. Тип: новая фича (затрагивает `apps/api` — orders/auth, `apps/web`).

## Контекст

Сейчас купить можно только залогинившись: `Order.userId` обязателен, корзина
(`/api/cart`) серверная по `userId`, все order-роуты и оплата (`create-order`/
`capture`/`claim`) — за `requireAuth` с проверкой `order.userId === userId`.
Для импульсной покупки хэндмейд-подарка ($16–28) обязательная регистрация — заметный
барьер: часть покупателей уходит на шаге «зарегистрируйтесь».

Платёжный путь только что стабилизирован (фикс invoice_id + webhook), и его трогать
повторно нежелательно.

## Цель

Покупатель оформляет и оплачивает заказ **без регистрации** — вводит email и адрес,
платит. Под капотом создаётся аккаунт по email (вариант «гость + авто-аккаунт»), так
что история заказов сохраняется и покупатель при желании позже входит (через Google
или задав пароль). Платёжный путь не переписывается.

## Модель данных

**Схема не меняется** — `User` уже готов: `passwordHash String?`, `googleId String?`,
`emailVerified Boolean`, `name String`.

**Гостевой аккаунт** = обычная запись `User` с `passwordHash = null` И `googleId = null`.
Отдельный флаг/таблица не нужны — «гостевой» выводится из отсутствия обоих credential'ов.

**Классификация email при оформлении** (основа безопасности):

| Состояние email | Признак | Поведение |
|---|---|---|
| не существует | — | создать гостевой `User` + выдать сессию |
| существует, `passwordHash==null && googleId==null` | прошлый гость, войти сам не может | переиспользовать + выдать сессию |
| существует, есть `passwordHash` ИЛИ `googleId` | настоящий аккаунт (умеет входить) | **сессию НЕ выдавать → 409 «войдите»** |

`Order` — без изменений: `userId` остаётся, гостевой и обычный заказ идентичны.

## Поток оформления

**Гостевая корзина — клиентская** (localStorage): серверная `/api/cart` живёт по
`userId`, которого у гостя нет. Залогиненный — серверная корзина как сейчас.

**Новый эндпоинт `POST /orders/guest`** (без `requireAuth`, с rate-limit). Принимает
`email`, `shippingAddress`, `items` (только `productId` + `quantity` + `message`, **без
цен**). Шаги:

1. **Пересчёт на сервере**: наличие, сток, цены, активная скидка, доставка. Цены из
   клиента не принимаются.
2. **find-or-create по email** + развилка из таблицы выше. Настоящий аккаунт → `409`,
   заказ не создаётся. Иначе создать/переиспользовать гостевого `User` (имя =
   `shippingAddress.fullName`).
3. **Создать заказ** — та же логика, что текущий `createOrderFromCart`, но из
   переданных items, а не из серверной корзины.
4. **Выдать сессию** — `issueTokens` ставит те же access/refresh cookies, что при входе.
5. Вернуть `{ orderId, orderNumber, amount }`.

**Дальше — существующий путь оплаты без изменений**: после шага 4 у гостя есть
полноценная сессия, поэтому `create-order` → `capture` → webhook → `markOrderPaid`
работают ровно как у залогиненного. Платёжку не трогаем.

```
гость: товары (localStorage) → Email + адрес → PayPal
  → POST /orders/guest
     ├─ email зарегистрирован → «войдите», форма входа
     └─ ок → сессия выдана, заказ создан → обычная оплата PayPal
```

## Безопасность

- **Захват чужого аккаунта** закрыт развилкой: настоящий email → `409`, сессия не
  выдаётся. Войти в чужой аккаунт вводом email нельзя.
- **Email enumeration** — осознанный компромисс: ответ «email зарегистрирован»
  косвенно раскрывает наличие аккаунта (как все формы входа). Смягчается rate-limit'ом;
  для текущего трафика приемлемо.
- **Верификация email не блокирует оформление** (иначе теряется смысл «без трения»).
  `emailVerified` остаётся `false`; реальная верификация — при задании пароля или входе
  через Google. Опечатка в email → заказ оформлен/оплачен, но письмо не дойдёт.
- **Rate-limit на `/orders/guest`** обязателен (эндпоинт без auth): анти-спам и
  анти-перебор. Тот же механизм, что на платёжных роутах.
- **Сумма считается только на сервере** — security-инвариант.
- **Присвоение гостевого аккаунта** при первом реальном входе владельца. Переиспользует
  `googleAuth` и `resetPassword`, НО два места проверить/поправить:
  - `googleAuth` должен **связать** существующий `User` по email (добавить `googleId`),
    не упасть на `unique email` и не создать дубль;
  - `requestPasswordReset`/`resetPassword` должны **задавать пароль впервые** аккаунту
    без `passwordHash`, а не только «сбрасывать» существующий.

## Фронтенд

- **Гостевая корзина** в localStorage (Pinia + персист); add-to-cart для гостя не зовёт
  серверный `/api/cart`. Слияние гостевой корзины с серверной при входе — опционально
  (не в первой итерации).
- **Поле Email** на странице корзины — обязательное, только для гостя.
- **Ветка в `prepareOrder`** (`CartPageWidget`): гость → `POST /orders/guest`; успех →
  сессия + `orderId` → обычная оплата; `409` → сообщение «у вас есть аккаунт, войдите» +
  форма/ссылка входа.
- **`PaypalPayment.vue` — без изменений** (после guest-checkout гость «залогинен»).
- Письмо-подтверждение заказа: необязательная ссылка «создать пароль» (backend).

## Тестирование

- **Backend unit (TDD):** guest-checkout — новый email → гость+заказ+сессия; настоящий
  email → `409`; гостевой существующий → переиспользование; пустая корзина → `400`; нет
  стока/недоступен → ошибка; подделка цены игнорируется. `googleAuth` связывает гостевой
  email без дубля. `resetPassword` задаёт пароль впервые. Роут `/orders/guest` доступен
  без auth; rate-limit срабатывает.
- **Integration:** полный guest-checkout до созданного заказа + выданной сессии.
- **Web unit:** гостевая ветка `CartPageWidget` (email-поле, `/orders/guest`, `409 →
  вход`); гостевая корзина в localStorage.
- **e2e (опционально):** гость оформляет и платит sandbox-покупателем.

## Объём и риски

- Новый use-case + роут `/orders/guest`, репозиторный метод «заказ из переданных items»,
  правки `googleAuth` (связывание) и password-reset (первичная установка), фронт
  (гостевая корзина + развилка оформления). Без изменений схемы БД и без правок
  платёжного пути.
- Риск №1 — **безопасность развилки email** (захват аккаунта); тест на `409` для
  настоящего аккаунта обязателен.
- Риск №2 — **аккуратность `googleAuth`/password-reset** при присвоении: не создать дубль
  по unique email, корректно задать первый пароль.
- Затрагивает auth → `code-reviewer` (critical/high блокируют) и полный прогон тестов
  обязательны.

## Что НЕ делаем (YAGNI)

- Не разделяем «покупатель» и «получатель» — имя берём из адреса доставки.
- Не блокируем оформление email-верификацией.
- Не сливаем гостевую корзину с серверной при входе в первой итерации (можно добавить позже).
- Не вводим отдельную модель «guest order» / токен заказа — авторизация остаётся по
  `userId` (это и есть смысл выбранного подхода A).
